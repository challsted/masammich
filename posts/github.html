What is Github?

Well, a simple google search can give us an answer to that:
GitHub is a Git repository web-based hosting service, which offers all of the distributed revision control and source code management (SCM) functionality of Git as well as adding its own features.

So, that might make since to some of you, but Ill admit, ive been useing github for almost a year as of this writeing, and out of that year, about the last 4 months ive used it A LOT, and even then, the Google deffinition is a little hard to understand. So I will attempt to put my own spin on what was writen above but with a lot more claification.

Github is a cloud based code repository which utalizes the version control language of "git". Ill get into what is "git" more soon, and what "version control" is also, but for now lets just talk about what Github actually is. Github IS a online, cloud based repository accessable by the public, but also has private repositorys. Github IS a place for developers and hackers alike to share code, talk about the code that is written or planned to be written, and utalize opensource

What is Git?

Another question you might be asking is Why is git better then SVN or CVS or any other similar language?
While I'll try to not show my obvious bias towards git in this explination here it goes:
With most other version control clients, there is a central repository somewhere, in the cloud or on a server at work, or on a vm in a rack stored in a closet you dont use much. That repository is the only repository there is, you might be able to branch from it, and push to branches, but this dosnt feel very normal, and its hard to use and keep updated. When you update code, it updates the master repository. Your only option if you want to have a test bed, is to make a copy of the origonal, and push to the copy for testing. Then when you want to add stable code to the main, you then copy the code over to a different folder struture, and push it up. Sure, it works, and it has worked well for many years, but its old and useless now a days.

As pointed out to me when researching this article, prior to github, if you wanted to contribute to an opensource project there was a huge pain in the but way for doing it.
First you had to download the origional source code (which possibly is behind what the developer is activly working on).
Go though the code till you find where the fix you want to implement needs to be added, test to make sure that the code your adding actually fixes what you want, and dosnt break anything else
Create whats known as a "patch" which lists all of your changes, then submit that patch to the owner(s)/maintaner(s) of the project.
They would then have to look though your patch, make sure its not melicious in any way
Patch a copy of there code with your patch (if they deemed it good)
Make sure it fixes what you say the patch is there to fix, make sure it dosnt break any other features they have then add it to what ever source code repo they have so they can push out your patch with the next release.

This can be a HUGE pain for the main developer, if its a popular project they might recieve 10-1,000 emails a day with patches, bug reports, complaints, legal issues, what ever it is. This takes away from the time they can actually develop. Also, you might be the 3rd person that week who has sent in a patch to the exact same issue you sent in. So then the developer has to chose WHO's patch to accept. Honestly, im not sure how some projects made it with all of that.

With GIT, you "clone" the repository to your local machine, you edit everything on your local machine, test to make sure its all good locally, then you push it back up to origin.
Branching feels normal, switching between branches makes since and can be easily followed, pushing branches back up to github (or where ever) feels good, and everyone can see whats going on in what branch.
The act of commiting (while can be worked around) is forced. While being forced to do something normaly would be a bad thing, in this case its not, when you commit code, you have to give a short (~100 or less characters) of what you did. So if you are an editor, and you fixed some spelling errors in the document, you can put "fixed spelling errors". Or if your a developer and you finally fixed that darn bug that keeps dumping your tables, you could comit "fixed bug 1c487"
If you want to contribute to an opensource project, but your not part of the developer team, you can fork their code, make the changes on your local machine, then submit whats known as a "pull" request back to the owner. This pull request can be reviewed by many eyes, or at least the owner's. He can quickly see what changes you made to his code because the system is built to do all of that for you. He can copy down your changes, make sure they work, then push them back in.
Does this remove the pain of dealing with other people fixing/updateing your code? Not at all, but it makes the whole process a lot easier, much more visual, and allows for more people to know whats going on.

Version control is the next big thing to explain. Version control is a history of the codes, adds and removes. For example, this blog post your viewing, is written on my local machine, every now and then I save. Every save is saved to a copy of the full repository on my local machine. Once i get to a point where i feel its got enough content to where if i lost it I would be angry, I push it up to a branch and in the commit i say WIP which is short for Work In Progress. Once im complete with all of what i think needs to be done, i commit it as "Review" and contact my editor and ask him to review the document. If hes happy with it he tells me so, if not he simply fixes what ever errors i have, or he asks me to work on certain areas. This continues until he says its good. I then "merge" that branch into the master. Merging a branch is a litle advanced for this paragraph but ill try to cover it soon or in a future article. For now, just know that it pulls all of the code (in this case text) i put into the branch into the master. Once i have a full new post, or what ever the goal was (fixed bug, new post, updated piece of code...) i then log onto the server, pull the code down, and copy the new parts over to the apache webserver. A few minutes later i refresh the live page and check it there to make sure it looks good.
So, thats great, sure, but what does thtat have to do with version control?
Lets say im fixing something, refactoring code, adding a feature, updateing a post, what ever it is. I commit, and i realize, wait a minute, this whole sectioin here is pointless now i just removed its need, or i refactored it. I commit that new piece up, everything looks good, but then we find out that the page dosnt work on Chrome, or when i refactored the code i added a bug that was worse then the origional bug or what ever. I can then roll back to a previous version of the code in a snap! Everyone who views my code can see exactly what lines i added, what lines i removed, maybe when i look at the old and new code that was changed side by side i can see what i did wrong, and fix it, or i can only revert back half of it.

This is the beauity of Github, Git and Version Control, and why everyone should use it. If your a developer, or if you just write scripts for personal scripts. Or if your like me, you run linux and you have a small collection of "dotfiles" (future article on these) and you want a place to store them, or a place to share them because you think yours are awesome. Github is the best place to store what ever it is your doing!

How do I start useing Github?

So, first, you will need to get git.
On linux, this is in every repository ive ever used, so it should be as simple as just running what ever your package manager's command to install something then "git". For example:
sudo apt-get install git <-- this will install git on Ubuntu/Debian Based Linux Distros
pacman -S git <-- this will install git on Arch Linux

You should also sign up for a free github account here(you only have to pay if you want private reositorys which is optional).
Once your in you need to actually setup your account.

CLI

So, we fire up a terminal as always and we type the following (green text = input, red text = output)
git config --global user.name "John Doe" Replace John Doe with your username you created on github! Keep the Quotation Marks though!
git config --global user.email "johndoe@foobar.com" Replace the email with your email! Keep the Quotation Marks though!
(Optional)git config --global core.editor "vim" By default it opens in Emacs you can also set it to Nano by useing "nano -w"
(Optional)git difftool --global --tool=vimdiff By default it uses "meld"

Now that your account is on your local machine, and you have git installed lets get a empty repositry started. Ill show you how preform this.

GUI Way (The way i always do it)

  1. Log into your account at github.com
  2. In the top right, to the right of your user name, click the "+", then click "New Repository"
  3. Type in a name for the repository
  4. (Optional) Give a description of what you plan on doing with the repository
  5. (Optional) Change the repo to be "private", this requries a paid account to do
  6. (Optional) Click the checkbox to add a "README" file (I always do this)
  7. (Optional) Create a .gitignore this is useful if you know there are certain files you want to put into your repo, that you wont want git to put into the repo. For example, i have a ".project" folder that is only on my local due to it will not benefit anyone out there to have this info. Its for a local only GUI program that im expereimenting with at the moment called "GitEye".
  8. (Optional) Give the repo a license. If you need to know more about licenses, there is a small "i" with a circle that takes you to a place that explains what each license does and is used for.
  9. Click "Create Repopository"

Now your repository is created, its on GitHub, and if you selected the option to create a README file its also available to pull down to your local imediatly. After i explain how to do this all via CLI ill explain how to clone the repo localally also.

Clone your new Repository

  1. If your not already, log into GitHub, and click on your username up in the top right (this is your profile)
  2. Click on your new repository, if you have a file here already your ready to clone your repo (if not you need to create one)
  3. On the right side, towards the middle or bottom, you will see "HTTPS Clone URL" click in this box and copy manually, or click the little copy icon to the right of that to copy to your clipboard
  4. From your terminal, create a folder (i always create a folder called "git" at my home) and go inside of the folder.
  5. Run the following command git clone https://github.com/user.name/repo.git replace the URL with the one you just copied

You now have a repository on your local machine, if you want to make sure your able to upload/download code as needed. Lets do a quick exercise.
touch file1.txt && cat "HELLO WORLD!" > file1.txt
This should create a file called file1.txt and put the words "HELLO WORLD!" into the file.
git add file1.txt
This adds the 1 file "file1" to be sent back to github, you can also specifiy multipul files at a time (file1, file2, file3).
You can also use git add . this will add everything in the current folder (and subfolders) not just 1 or file.
git commit -m "Testing Access"
This preforms a "commit" you are commiting a the files in that are curerntly part of the "add" but not yet "commited" with the following message (-m) if you dont add -m and a message it will pop up a box asking you to type your commit message. If you set the optonal core.editor above it will use the editor you chose, other wise it uses emacs.
git push
This "pushes" everything that is currently commited back to where you cloned it (in this case "master")

Assuming you dont see any errors preforming the above, lets go back to github, and refresh our repository, if you see that "file1.txt" is now listed inside of the repo, then contradulations! It worked!

If not you need to troubleshoot. You can send me an email if needed via the contact options above, ill try to help you. But beware i do all of this in my sparetime, so im not sure how fast i will be able to reply. Make sure you put an email you can check into the email box also.

Ok, now what?

Now you can start makeing your own stuff! If your already a developer in any language, you should already be pretty excited, if your just getting your feet wet in this world then your probably meaning this more litterly. Lets go over 2 very important things you can do in GitHub/Git that are VERY useful. Cloneing and Forking.

Cloneing

git clone https://github.com/user.name/repo.git
OR
git clone git@github.com:user.name/repo.git
If you have setup ssh access to github, this is a quick and easy way to preform this.

If you followed the above, you have already cloned a empty repository, added a file, and reuploaded it back to your github all in a few lines of code.
Cloneing(in git) is the act of copying the current set of code from any public repository on the web to your local machine, you dont have to have an account on github to clone someone, you just need to have git installed! The command is git clone repository
As i suggested above, you should probably create a folder to host all of the clones you make in, my suggestion is to do it in a place you are frequently at, so on linux its your "home" directory, and i suggest your folder be called "git", "Git", "GIT" or if you want it slightly hidden name it ".git".

Something cool you can do if you can search github, and it will return public repositorys. You can start the serach by typeing what ever words you want to appear in the repo or description. Then if you want to change it to only a certain language, you can click a desiered language on the left, you can change the sort option to see the repo thats has been followed the most, or started the most (or what ever option you want). Then when you get something you might like, you can open up the repo and take a look around. If you like what you see, and want to play with it on your local machine, simply clone it. If you want to take a copy of this for your own to edit, you need to fork it (next section).

The main thing i use cloneing for, is if i want to have local copie(s) of my code on a local machine, or on a server or my RPi. You can pre-write the little clone command, and give it to friends so they can simply copy pasta and get your code. You can use it in scripts to copy down code from GH, execute something against it, then delete it, or what ever you need. Its a very simple, yet very awesome little command.

Forking

Forking is used to make a copy of someone else's code into your profile so that you can edit it as needed with out hurting the origional project.
Due to this is only done via a web-gui (To my knowledge) i cant provide you any code examples...
BUT I can give you a repo that's only purpose is to fork!
If you go over to the octocat profile and go to the "spoon-knife" repo (HERE)
You can click "fork" in the top right, this will then give you a funn litle animation of a book being photo copied then it will forward you back to your profile, but with the new repo (in this case Spoon-Knife) selected! Now you can clone it, make edits, and upload it back as you please. It wont effect the origional repo owned by "Octocat".

Now, lets say that you were forking a project, with the intention of pushign it back to the master repo! You would do this for a few reasons, 1 of which is due to you have tried your hand at patching something, or at adding a feature, but you are not one of the colaborators.
After you fork their repo, make the changes you need to, push the code back to your forked repo on GH. You can go to the origional repo, and click "create pull request". It will ask you to put in comments on what your doing. This is where you say "i added X feature" or "Fixed Y bug" its best if you identify bugs by the issued code if its a known bug. You should be able to get this information from the "issues" tab (im assumging you know what im talking about if your contributeing to opensource projects right now). Then the owner of the repo (when they have time) can look though your changes, and if they feel its good enough, they can "merge" it back in. If this happens, congradulations! You made a chagne to someone else's code :D
