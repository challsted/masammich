What is Github?

Well, a simple google search can give us an answer to that:
GitHub is a Git repository web-based hosting service, which offers all of the distributed revision control and source code management (SCM) functionality of Git as well as adding its own features.

So, that might make since to some of you, but Ill admit, ive been useing github for almost a year as of this writeing, and out of that year, about the last 4 months ive used it A LOT, and even then, the Google deffinition is a little hard to understand. So I will attempt to put my own spin on what was writen above but with a lot more claification.

Github is a cloud based code repository which utalizes the version control language of "git". Ill get into what is "git" more soon, and what "version control" is also, but for now lets just talk about what Github actually is. Github IS a online, cloud based repository accessable by the public, but also has private repositorys. Github IS a place for developers and hackers alike to share code, talk about the code that is written or planned to be written, and utalize opensource

What is Git?

Another question you might be asking is Why is git better then SVN or CVS or any other similar language?
While I'll try to not show my obvious bias towards git in this explination here it goes:
With most other version control clients, there is a central repository somewhere, in the cloud or on a server at work, or on a vm in a rack stored in a closet you dont use much. That repository is the only repository there is, you might be able to branch from it, and push to branches, but this dosnt feel very normal, and its hard to use and keep updated. When you update code, it updates the master repository. Your only option if you want to have a test bed, is to make a copy of the origonal, and push to the copy for testing. Then when you want to add stable code to the main, you then copy the code over to a different folder struture, and push it up. Sure, it works, and it has worked well for many years, but its old and useless now a days.
With GIT, you "clone" the repository to your local machine, you edit everything on your local machine, test to make sure its all good locally, then you push it back up to origin.
Branching feels normal, switching between branches makes since and can be easily followed, pushing branches back up to github (or where ever) feels good, and everyone can see whats going on in what branch.
The act of commiting (while can be worked around) is forced. While being forced to do something normaly would be a bad thing, in this case its not, when you commit code, you have to give a short (~100 or less characters) of what you did. So if you are an editor, and you fixed some spelling errors in the document, you can put "fixed spelling errors". Or if your a developer and you finally fixed that darn bug that keeps dumping your tables, you could comit "fixed bug 1c487"

Version control is the next big thing to explain. Version control is a history of the codes, adds and removes. For example, this blog post your viewing, is written on my local machine, every now and then I save. Every save is saved to a copy of the full repository on my local machine. Once i get to a point where i feel its got enough content to where if i lost it I would be angry, I push it up to a branch and in the commit i say WIP which is short for Work In Progress. Once im complete with all of what i think needs to be done, i commit it as "Review" and contact my editor and ask him to review the document. If hes happy with it he tells me so, if not he simply fixes what ever errors i have, or he asks me to work on certain areas. This continues until he says its good. I then "merge" that branch into the master. Merging a branch is a litle advanced for this paragraph but ill try to cover it soon or in a future article. For now, just know that it pulls all of the code (in this case text) i put into the branch into the master. Once i have a full new post, or what ever the goal was (fixed bug, new post, updated piece of code...) i then log onto the server, pull the code down, and copy the new parts over to the apache webserver. A few minutes later i refresh the live page and check it there to make sure it looks good.
So, thats great, sure, but what does thtat have to do with version control?
Lets say im fixing something, refactoring code, adding a feature, updateing a post, what ever it is. I commit, and i realize, wait a minute, this whole sectioin here is pointless now i just removed its need, or i refactored it. I commit that new piece up, everything looks good, but then we find out that the page dosnt work on Chrome, or when i refactored the code i added a bug that was worse then the origional bug or what ever. I can then roll back to a previous version of the code in a snap! Everyone who views my code can see exactly what lines i added, what lines i removed, maybe when i look at the old and new code that was changed side by side i can see what i did wrong, and fix it, or i can only revert back half of it.

This is the beauity of Github, Git and Version Control, and why everyone should use it. If your a developer, or if you just write scripts for personal scripts. Or if your like me, you run linux and you have a small collection of "dotfiles" (future article on these) and you want a place to store them, or a place to share them because you think yours are awesome. Github is the best place to store what ever it is your doing!
